# 04 — 数据源与方言

## 支持的 8 种数据源

| 数据源 | 类型 | 驱动 | 协议 | 典型场景 |
|--------|------|------|------|---------|
| DuckDB | 嵌入式分析 | `duckdb` | 内嵌 | 本地分析、示例数据、CSV 查询 |
| MySQL | 事务型 | `aiomysql` | TCP 3306 | 业务库、订单系统、用户数据 |
| PostgreSQL | 混合型 | `asyncpg` | TCP 5432 | 分析+事务混合、JSON 查询 |
| ClickHouse | 列式分析 | `clickhouse-connect` | HTTP 8123 | 大规模分析、日志数据、实时看板 |
| Doris | MPP 分析 | `aiomysql` | TCP 9030 | 多维分析、数据仓库（MySQL 协议兼容） |
| Hive | 离线分析 | `pyhive` | Thrift 10000 | 大数据离线报表、T+1 分析 |
| Presto | 联邦查询 | `pyhive` | TCP 8080 | 跨数据源联邦查询 |
| Spark | 大数据计算 | `pyhive` | Thrift 10001 | 批处理、大规模 ETL |

## SQL 方言差异对照表

### 分页

| 数据源 | 语法 | 示例 |
|--------|------|------|
| MySQL / Doris | `LIMIT n OFFSET m` | `SELECT * FROM t LIMIT 10 OFFSET 20` |
| PostgreSQL | `LIMIT n OFFSET m` | 同上 |
| ClickHouse | `LIMIT n OFFSET m` | 同上 |
| SQL Server | `OFFSET m ROWS FETCH NEXT n ROWS ONLY` | `SELECT * FROM t ORDER BY id OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY` |
| Hive | `LIMIT n`（不支持 OFFSET） | `SELECT * FROM t LIMIT 10` |
| Presto | `LIMIT n OFFSET m` | 同 MySQL |
| Spark | `LIMIT n` | `SELECT * FROM t LIMIT 10` |

### 字符串拼接

| 数据源 | 语法 | 示例 |
|--------|------|------|
| MySQL | `CONCAT(a, b)` | `CONCAT(first_name, ' ', last_name)` |
| PostgreSQL | `a \|\| b` | `first_name \|\| ' ' \|\| last_name` |
| ClickHouse | `concat(a, b)` | `concat(first_name, ' ', last_name)` |
| Hive | `CONCAT(a, b)` | 同 MySQL |
| SQL Server | `a + b` | `first_name + ' ' + last_name` |

### 日期格式化

| 数据源 | 语法 | 示例 |
|--------|------|------|
| MySQL | `DATE_FORMAT(col, '%Y-%m-%d')` | `DATE_FORMAT(create_time, '%Y-%m')` |
| PostgreSQL | `TO_CHAR(col, 'YYYY-MM-DD')` | `TO_CHAR(create_time, 'YYYY-MM')` |
| ClickHouse | `formatDateTime(col, '%Y-%m-%d')` | `formatDateTime(create_time, '%Y-%m')` |
| Hive | `DATE_FORMAT(col, 'yyyy-MM-dd')` | `DATE_FORMAT(create_time, 'yyyy-MM')` |
| Presto | `DATE_FORMAT(col, '%Y-%m-%d')` | 同 MySQL |

### 日期运算

| 数据源 | 语法 | 示例 |
|--------|------|------|
| MySQL | `DATE_SUB(NOW(), INTERVAL 7 DAY)` | 最近7天 |
| PostgreSQL | `CURRENT_DATE - INTERVAL '7 days'` | 最近7天 |
| ClickHouse | `date_sub(day, 7, today())` | 最近7天 |
| Hive | `DATE_SUB(CURRENT_DATE(), 7)` | 最近7天 |
| Presto | `DATE_ADD('day', -7, CURRENT_DATE)` | 最近7天 |

### 类型转换

| 数据源 | 语法 | 示例 |
|--------|------|------|
| MySQL | `CAST(col AS类型)` | `CAST(amount AS DECIMAL(10,2))` |

| ClickHouse | `CAST(col AS类型)` | 同 MySQL |
| Hive | `CAST(col AS类型)` | 同 MySQL |

## 方言适配策略

ChatSQL 采用 **Agent 动态适配** 策略，而非静态 SQL 转译：

```
1. System Prompt 注入目标库的方言规则
2. Agent 根据方言规则生成对应 SQL
3. 如果 Agent 不确定，使用最通用的 ANSI SQL 语法
```

### 为什么不用 SQL 转译器？

| 方案 | 优点 | 缺点 |
|------|------|------|
| SQL 转译器（如 JSqlParser） | 精确转换 | 覆盖不全、维护成本高、新函数难适配 |
| **Agent 动态生成** | 灵活、可处理任意方言、LLM 天然理解 SQL 方言 | 偶尔生成错误 SQL |

ChatSQL 选择 Agent 动态生成，因为：
1. LLM 本身理解多种 SQL 方言
2. System Prompt 注入方言规则后准确率很高
3. 三层防线兜底（语法校验 → 执行验证 → 用户确认）
4. 学习系统会积累方言经验，越用越准

## 元数据自动获取

配置数据源后，ChatSQL 自动获取表结构并注入 System Prompt：

### 获取内容

| 元数据 | 说明 | 用途 |
|--------|------|------|
| 表名 | 所有表的名称 | Agent 知道有哪些表可用 |
| 字段名 | 每个表的列名 | Agent 知道怎么写 SELECT |
| 字段类型 | VARCHAR / INT / TIMESTAMP 等 | Agent 知道怎么写 WHERE 条件 |
| 注释 | 字段和表的 COMMENT | Agent 理解业务含义 |
| 采样数据 | 前几行数据（可选） | Agent 理解数据格式和值域 |

### 注入方式

```
DataSource.describe_table("orders")
    ↓
返回: [
  {name: "order_id", type: "VARCHAR", comment: "订单号"},
  {name: "status", type: "VARCHAR", comment: "状态"},
  ...
]
    ↓
ResponsesService._build_system_prompt()
    ↓
注入 System Prompt 的"可用表结构"段
```

### 自动刷新

- 每次对话时重新获取元数据
- 新增表/字段自动可见，无需重启
- 支持手动刷新（设置面板 → 数据源 → 查看元数据）
