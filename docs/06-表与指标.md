# 06 — 表与指标

## 元数据如何注入 System Prompt

Agent 的"知识来源"是 System Prompt 中的元数据，而非硬编码。

```
DataSourceManager.get_full_metadata()
    ↓
返回所有表的结构信息
    ↓
ResponsesService._build_system_prompt()
    ↓
格式化为自然语言描述，注入 System Prompt
```

### 注入内容

```markdown
## 可用表结构

### orders（物流订单）
字段 | 类型 | 说明
order_id | VARCHAR | 订单号
status | VARCHAR | 状态：已揽收/运输中/已签收
sort_center | VARCHAR | 分拣中心编码
create_time | TIMESTAMP | 创建时间
amount | DECIMAL | 金额

### sort_center（分拣中心）
字段 | 类型 | 说明
center_id | VARCHAR | 中心编码
center_name | VARCHAR | 中心名称
region | VARCHAR | 所属区域
```

### Agent 从元数据中学到什么

| 元数据 | Agent 学到的 | 影响 |
|--------|-------------|------|
| 表名 | 有哪些表可用 | 决定 FROM 哪个表 |
| 字段名 | 每个表有哪些列 | 决定 SELECT 什么、WHERE 什么 |
| 字段类型 | 列的数据类型 | 决定怎么写条件（字符串要引号、数值不要） |
| 注释 | 业务含义 | 理解"订单量"= COUNT(order_id) |
| 采样数据 | 实际值长什么样 | 理解 status 的值域（已揽收/运输中/已签收） |

## 如何区分表

### 按业务域

| 业务域 | 典型表 | 说明 |
|--------|--------|------|
| 订单 | orders, order_detail, order_log | 订单主表、明细、日志 |
| 物流 | shipments, routes, tracking | 运单、路线、轨迹 |
| 分拣 | sort_center, sort_task | 分拣中心、分拣任务 |
| 用户 | users, addresses, contacts | 用户、地址、联系方式 |
| 财务 | payments, invoices, refunds | 支付、发票、退款 |

### 按粒度

| 粒度 | 特征 | 示例 | 用途 |
|------|------|------|------|
| 明细表 | 每行一条记录 | orders | 查具体记录、钻取详情 |
| 汇总表 | 每行一个聚合值 | daily_order_stats | 快速查指标、仪表盘 |
| 维度表 | 描述性信息 | sort_center | 关联查询、翻译编码 |

### 按数据源

当多个数据源有同名表时，通过数据源名称区分：

```
MySQL.orders — 业务订单
ClickHouse.orders — 日志中的订单记录
```

Agent 在 System Prompt 中看到的是按数据源分组的元数据，不会混淆。

## 指标计算规则

### 基础聚合

| 指标 | SQL | 说明 |
|------|-----|------|
| 总数 | `COUNT(*)` | 所有行数 |
| 去重计数 | `COUNT(DISTINCT col)` | 去重后的数量 |
| 求和 | `SUM(col)` | 数值列求和 |
| 平均 | `AVG(col)` | 数值列平均值 |
| 最大 | `MAX(col)` | 最大值 |
| 最小 | `MIN(col)` | 最小值 |

### 复合指标

| 指标 | 公式 | SQL |
|------|------|-----|
| 转化率 | 成交数 / 访问数 | `COUNT(CASE WHEN status='已签收' THEN 1 END) * 100.0 / COUNT(*)` |
| 客单价 | 总金额 / 订单数 | `SUM(amount) / COUNT(DISTINCT order_id)` |
| 完成率 | 完成数 / 总数 | `COUNT(CASE WHEN status='已完成' THEN 1 END) * 100.0 / COUNT(*)` |
| 占比 | 部分 / 总体 | `SUM(CASE WHEN region='华东' THEN amount END) * 100.0 / SUM(amount)` |

### 同比/环比

**同比**（去年同期）：
```sql
-- 月度同比
SELECT
  DATE_FORMAT(date, '%Y-%m') AS month,
  SUM(amount) AS current,
  LAG(SUM(amount), 12) OVER (ORDER BY DATE_FORMAT(date, '%Y-%m')) AS last_year,
  (SUM(amount) - LAG(SUM(amount), 12) OVER (ORDER BY DATE_FORMAT(date, '%Y-%m')))
    * 100.0 / LAG(SUM(amount), 12) OVER (ORDER BY DATE_FORMAT(date, '%Y-%m')) AS yoy_rate
FROM orders
GROUP BY DATE_FORMAT(date, '%Y-%m')
ORDER BY month;
```

**环比**（上一期）：
```sql
-- 周度环比
SELECT
  DATE_FORMAT(date, '%x-W%v') AS week,
  SUM(amount) AS current,
  LAG(SUM(amount), 1) OVER (ORDER BY DATE_FORMAT(date, '%x-W%v')) AS last_week,
  (SUM(amount) - LAG(SUM(amount), 1) OVER (ORDER BY DATE_FORMAT(date, '%x-W%v')))
    * 100.0 / LAG(SUM(amount), 1) OVER (ORDER BY DATE_FORMAT(date, '%x-W%v')) AS wow_rate
FROM orders
GROUP BY DATE_FORMAT(date, '%x-W%v')
ORDER BY week;
```

> **注意**：同比/环比依赖窗口函数，MySQL 5.7 及以下不支持。Agent 会根据方言自动调整写法。

### 自定义指标扩展

用户可以通过学习系统让 Agent 记住自定义指标：

```
用户: "我们的签收率 = 已签收订单数 / 总订单数"
Agent: 学习并记住这个定义
下次用户: "查一下签收率"
Agent: 自动使用已签收订单数 / 总订单数的公式
```

## 表关系与 JOIN 策略

### 关系发现

Agent 通过以下方式发现表之间的关系：

1. **字段名匹配** — `orders.sort_center` = `sort_center.center_id`
2. **注释线索** — 字段注释中提到关联表
3. **外键约束** — 如果数据库定义了外键
4. **学习经验** — 历史对话中学到的关联方式

### JOIN 策略

| 场景 | JOIN 类型 | 示例 |
|------|----------|------|
| 订单+分拣中心名称 | INNER JOIN | `orders JOIN sort_center ON orders.sort_center = sort_center.center_id` |
| 所有订单（含无中心的） | LEFT JOIN | `orders LEFT JOIN sort_center ON ...` |
| 仅关联成功的 | INNER JOIN | `orders INNER JOIN sort_center ON ...` |

### Agent 自动选择 JOIN 类型

```
用户: "查一下每个分拣中心的订单量"
Agent 分析:
  - 需要 orders 表（订单数据）
  - 需要 sort_center 表（中心名称）
  - 关联字段: orders.sort_center = sort_center.center_id
  - 选择 LEFT JOIN（保留没有订单的中心）
生成:
  SELECT sc.center_name, COUNT(o.order_id) AS order_cnt
  FROM sort_center sc
  LEFT JOIN orders o ON sc.center_id = o.sort_center
  GROUP BY sc.center_name;
```
