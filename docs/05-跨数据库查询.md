# 05 — 跨数据库查询

## 什么场景需要跨库查询

| 场景 | 说明 | 示例 |
|------|------|------|
| 业务库 + 分析库 | 事务数据在 MySQL，分析数据在 ClickHouse | 订单在 MySQL，用户画像在 ClickHouse |
| 多租户同结构 | 不同租户的数据在不同库，结构相同 | 客户 A 的 MySQL 库 + 客户 B 的 MySQL 库 |
| 维度表 + 事实表分离 | 维度表在 A 库，事实表在 B 库 | 维度表在 PostgreSQL，事实表在 Doris |
| 跨系统关联 | 不同业务系统的数据需要联合分析 | ERP 的订单 + WMS 的库存 |

## 跨库查询架构

```
用户: "对比 MySQL 的订单量和 ClickHouse 的日志量"

    ┌─────────────────────────────────────┐
    │           Agent 决策层               │
    │                                     │
    │  识别到需要查两个数据源               │
    │  生成两条独立 SQL                    │
    └──────────┬──────────┬───────────────┘
               │          │
    ┌──────────▼──┐  ┌────▼──────────┐
    │  MySQL      │  │  ClickHouse   │
    │  SELECT ... │  │  SELECT ...   │
    │  结果 A     │  │  结果 B       │
    └──────────┬──┘  └────┬──────────┘
               │          │
    ┌──────────▼──────────▼───────────────┐
    │         应用层合并                    │
    │                                     │
    │  1. 列对齐（相同字段名对齐）          │
    │  2. 类型归一化（VARCHAR/STRING 统一） │
    │  3. 行合并（UNION 或 JOIN）          │
    │  4. NULL 填充（缺失字段补默认值）     │
    └──────────────┬──────────────────────┘
                   │
    ┌──────────────▼──────────────────────┐
    │         返回合并结果                  │
    │  columns: [date, source, count]     │
    │  rows: [                            │
    │    {date: "06-15", source: "订单", count: 100}, │
    │    {date: "06-15", source: "日志", count: 5000} │
    │  ]                                  │
    └─────────────────────────────────────┘
```

ChatSQL 采用 **方案一：应用层合并**，原因：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **应用层合并**（ChatSQL 采用） | 灵活、无需额外引擎、支持任意数据源组合 | 大数据量时内存压力大 |
| 联邦引擎（Presto/Trino） | 统一 SQL、性能好 | 需要额外部署、配置复杂 |

## 结果拼接策略

### 1. 列合并（相同字段名自动对齐）

```
MySQL 结果:      ClickHouse 结果:
| date    | cnt | | date    | cnt |
| 06-15   | 100 | | 06-15   | 500 |
| 06-16   | 120 | | 06-16   | 480 |

合并后（UNION 语义）:
| date    | cnt |
| 06-15   | 100 |
| 06-16   | 120 |
| 06-15   | 500 |
| 06-16   | 480 |
```

### 2. 行合并（相同结构表 UNION ALL）

当两个数据源的表结构相同时，直接行合并：

```python
def merge_rows(results: list[dict]) -> dict:
    """合并多个数据源的结果"""
    all_columns = []
    all_rows = []
    seen_cols = set()

    for result in results:
        for col in result["columns"]:
            if col["name"] not in seen_cols:
                all_columns.append(col)
                seen_cols.add(col["name"])
        all_rows.extend(result["rows"])

    # 缺失字段填充 NULL
    for row in all_rows:
        for col in all_columns:
            if col["name"] not in row:
                row[col["name"]] = None

    return {"columns": all_columns, "rows": all_rows}
```

### 3. 关联合并（应用层 JOIN）

当需要按某个维度关联两个数据源的结果时：

```
MySQL: 各分拣中心订单量
| center | order_cnt |
| A      | 100       |
| B      | 200       |

ClickHouse: 各分拣中心日志量
| center | log_cnt |
| A      | 5000    |
| B      | 8000    |

合并后（按 center 关联）:
| center | order_cnt | log_cnt |
| A      | 100       | 5000    |
| B      | 200       | 8000    |
```

```python
def join_results(result_a: dict, result_b: dict, join_key: str) -> dict:
    """按公共维度字段关联两个结果"""
    # 构建 B 的索引
    b_index = {}
    for row in result_b["rows"]:
        key = row.get(join_key)
        if key:
            b_index[key] = row

    # 合并
    merged_rows = []
    for row_a in result_a["rows"]:
        key = row_a.get(join_key)
        row_b = b_index.get(key, {})
        merged_row = {**row_a}
        # 添加 B 的非重叠字段
        for col in result_b["columns"]:
            if col["name"] != join_key and col["name"] not in row_a:
                merged_row[col["name"]] = row_b.get(col["name"])
        merged_rows.append(merged_row)

    # 合并列
    seen = {c["name"] for c in result_a["columns"]}
    new_cols = [c for c in result_b["columns"] if c["name"] not in seen]

    return {
        "columns": result_a["columns"] + new_cols,
        "rows": merged_rows,
    }
```

### 4. 类型归一化

不同数据库对同一类型的表示不同，需要统一：

| MySQL | PostgreSQL | ClickHouse | 归一化 |
|-------|-----------|------------|--------|
| VARCHAR | TEXT | String | string |
| INT | INTEGER | Int32 | integer |
| DECIMAL | NUMERIC | Decimal | decimal |
| DATETIME | TIMESTAMP | DateTime | datetime |
| DATE | DATE | Date | date |

### 5. NULL 处理

缺失字段统一填充：
- 字符串类型 → `""`（空字符串）
- 数值类型 → `0` 或 `NULL`
- 日期类型 → `NULL`
- 布尔类型 → `false`

## 分页与排序

### 各库分页差异

| 数据源 | 分页语法 |
|--------|---------|
| MySQL | `LIMIT 10 OFFSET 20` |
| PostgreSQL | `LIMIT 10 OFFSET 20` |
| ClickHouse | `LIMIT 10 OFFSET 20` |
| Hive | `LIMIT 10`（不支持 OFFSET） |
| Presto | `LIMIT 10 OFFSET 20` |
| Doris | `LIMIT 10 OFFSET 20` |

### 应用层统一分页

跨库查询时，不依赖各库的分页语法，而是在应用层统一分页：

```python
def paginate(rows: list, page: int, page_size: int) -> list:
    """应用层分页"""
    start = (page - 1) * page_size
    return rows[start : start + page_size]

def sort_rows(rows: list, sort_key: str, desc: bool = True) -> list:
    """应用层排序"""
    return sorted(rows, key=lambda r: r.get(sort_key, 0), reverse=desc)
```

### 大数据量场景

当单个数据源返回大量数据时：
1. 限制每个数据源最多返回 N 条（如 10000 条）
2. 应用层合并后内存排序
3. 超过阈值时提示用户缩小查询范围

## 跨库查询的 System Prompt 指令

当检测到用户问题涉及多个数据源时，System Prompt 会追加：

```
## 跨库查询规则
当前问题涉及多个数据源：
- MySQL (orders_db): 订单数据
- ClickHouse (logs): 日志数据

请分别生成两条 SQL，每条 SQL 使用对应数据源的方言。
结果将由系统自动合并。
请确保两个查询的分组字段名相同，以便系统关联。
```
