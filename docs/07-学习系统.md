# 07 — 学习系统

## 解决什么问题

| 问题 | 没有学习系统 | 有学习系统 |
|------|------------|-----------|
| 用户每次说"订单量" | Agent 每次都猜是 COUNT(*) | 记住用户习惯 COUNT(DISTINCT order_id) |
| 复杂指标如"签收率" | 每次都要用户解释公式 | 一次解释，永久记住 |
| 方言特殊写法 | Agent 可能写错 | 记住正确的方言写法 |
| 表关联方式 | 每次猜测 JOIN 条件 | 记住常用的关联关系 |

**核心思路**：不是训练模型，是从对话中积累经验规则。

## 三层防线（安全机制）

学习系统的核心安全设计。任何一条经验要被学习，必须通过全部三层防线。

```
对话产生
    │
    ▼
┌─────────────────────────────────┐
│  第一层：语法校验                  │
│                                 │
│  □ SQL 解析器能解析成功吗？        │
│  □ 关键字、表名、字段名合法吗？     │
│  □ 没有注入攻击风险吗？           │
│                                 │
│  不通过 → 直接拒绝，不学习         │
└──────────────┬──────────────────┘
               │ 通过
               ▼
┌─────────────────────────────────┐
│  第二层：执行验证                  │
│                                 │
│  □ SQL 实际执行成功吗？           │
│  □ 结果非空吗？                   │
│  □ 数值合理吗（无异常溢出）？       │
│  □ 时间范围符合预期吗？           │
│                                 │
│  不通过 → 直接拒绝，不学习         │
└──────────────┬──────────────────┘
               │ 通过
               ▼
┌─────────────────────────────────┐
│  第三层：用户反馈                  │
│                                 │
│  □ 用户确认结果正确吗？           │
│  □ 用户没有指出错误吗？           │
│                                 │
│  有些语义错误的 SQL 能跑通，       │
│  但结果是错的。大模型发现不了      │
│  的问题，靠人兜底。              │
│                                 │
│  不通过 → 直接拒绝，不学习         │
└──────────────┬──────────────────┘
               │ 通过
               ▼
         进入评分机制
```

### 为什么要三层防线？

| 层 | 防什么 | 示例 |
|----|--------|------|
| 语法校验 | 垃圾 SQL 被学进去 | `SELEC * FORM orders`（拼写错误） |
| 执行验证 | 能跑但没结果的 SQL | 查了一个空表，学了也没用 |
| 用户反馈 | 能跑但结果错的 SQL | `COUNT(*)` 而非 `COUNT(DISTINCT order_id)`，语义不同但语法正确 |

## 学习准入门槛

**三个条件必须全部满足，缺一不学：**

```
✅ SQL 执行成功        — 语法正确、数据库能跑
✅ 结果非空            — 查询有实际数据返回
✅ 用户确认正确        — 人说 OK 才算 OK
```

**核心原则：宁可少学一个对的，不要学一个错的。**

错误的经验比没有经验更危险——它会让 Agent 一直犯同样的错。

## 评分机制（五级分类）

通过三层防线后，进入评分机制，决定这条经验怎么处理：

```
新经验进入
    │
    ▼
┌─────────────────────────────────┐
│  与已有经验对比                    │
│                                 │
│  □ 完全重复？                     │
│  □ 全新模式？                     │
│  □ 有重叠但互补？                 │
│  □ 是已有经验的更好版本？          │
│  □ 是临时性规则？                 │
└──────────────┬──────────────────┘
               │
               ▼
    ┌──────────┼──────────┬──────────┬──────────┐
    │          │          │          │          │
    ▼          ▼          ▼          ▼          ▼
  SKIP     WRITE_NEW    MERGE    REPLACE   TTL_SHORT
```

### SKIP — 跳过

**条件**：已有相同或更高优先级的套路

**行为**：不写入，忽略

**示例**：
```
已有规则: "订单量 = COUNT(DISTINCT order_id)"
新经验: "订单量 = COUNT(DISTINCT order_id)"
→ SKIP（完全重复）
```

### WRITE_NEW — 写入新规则

**条件**：全新模式，且质量高

**行为**：创建新的 learned_routine 记录

**示例**：
```
已有规则: 无关于"签收率"的规则
新经验: "签收率 = COUNT(CASE WHEN status='已签收' THEN 1 END) * 100.0 / COUNT(*)"
→ WRITE_NEW（全新且正确）
```

### MERGE — 合并到已有规则

**条件**：与已有规则有重叠但互补

**行为**：将新经验合并到已有规则中，扩展规则的适用范围

**示例**：
```
已有规则: "订单量 = COUNT(*) — 按分拣中心统计"
新经验: "订单量 = COUNT(DISTINCT order_id) — 按时间范围统计"
→ MERGE（合并为：订单量 = COUNT(DISTINCT order_id)，支持按分拣中心和时间范围统计）
```

### REPLACE — 覆盖旧版

**条件**：新版本更优，旧版本过时或有误

**行为**：用新经验替换旧规则，旧规则标记为 replaced

**示例**：
```
已有规则: "订单量 = COUNT(*)"
新经验: "订单量 = COUNT(DISTINCT order_id)"（用户确认这才是正确的）
→ REPLACE（新版本更准确，覆盖旧版）
```

### TTL_SHORT — 短生命周期

**条件**：临时性规则（如特定活动期间的特殊统计口径）

**行为**：写入并设置短 TTL（如 7 天），到期自动清理

**示例**：
```
新经验: "双11期间订单量包含预售订单"
→ TTL_SHORT（临时规则，活动结束自动过期）
```

### 评分决策表

| 判断维度 | SKIP | WRITE_NEW | MERGE | REPLACE | TTL_SHORT |
|---------|------|-----------|-------|---------|-----------|
| 是否重复 | ✅ 完全重复 | ❌ 全新 | 部分重叠 | 版本升级 | 临时性 |
| 质量 | — | 高 | 中高 | 更高 | 中 |
| 持久性 | — | 永久 | 永久 | 永久 | 短期 |
| 影响 | 无 | 新增 | 扩展 | 替换 | 临时覆盖 |

## 淘汰机制

```
learned_routine 写入
    │
    ▼
  被查询匹配时 → 命中计数 +1，更新 last_hit_at
    │
    ▼
  长期未命中？
    │
    ├── 30天未命中 → 权重降低（排名靠后）
    ├── 60天未命中 → 标记为"不活跃"
    └── 90天未命中 → 自动淘汰（标记为 archived）
    │
    ▼
  错误的规则？
    │
    └── 被 REPLACE 覆盖 → 旧规则标记为 replaced
    │
    ▼
  TTL_SHORT 到期？
    │
    └── 自动清理（标记为 expired）
```

### 淘汰优先级

```
TTL_SHORT 到期 → 立即清理
错误被 REPLACE → 立即替换
长期未命中 → 逐步降权 → 最终淘汰
```

## Learn V2 数据模型

### learned_routines（已学习的模式）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| pattern | TEXT | 匹配模式（关键词/意图描述） |
| routine_type | VARCHAR | 类型：sql_pattern / metric_def / join_rule / dialect_rule |
| content | TEXT | 规则内容（SQL 模板/指标公式/JOIN 条件） |
| score | FLOAT | 质量分数（0-1） |
| status | VARCHAR | active / replaced / archived / expired |
| ttl | INTEGER | 生命周期（秒），NULL 为永久 |
| hit_count | INTEGER | 命中次数 |
| last_hit_at | TIMESTAMP | 最后命中时间 |
| created_at | TIMESTAMP | 创建时间 |
| source_session_id | VARCHAR | 来源会话 |

### learned_routine_audits（审计日志）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| routine_id | UUID | 关联的 routine |
| action | VARCHAR | WRITE_NEW / MERGE / REPLACE / SKIP / TTL_SHORT |
| reason | TEXT | 决策原因 |
| old_content | TEXT | 替换前的内容（REPLACE 时） |
| created_at | TIMESTAMP | 操作时间 |

### learn_tasks（学习任务）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| session_id | VARCHAR | 来源会话 |
| user_query | TEXT | 用户问题 |
| generated_sql | TEXT | 生成的 SQL |
| execution_result | TEXT | 执行结果摘要 |
| user_confirmed | BOOLEAN | 用户是否确认正确 |
| status | VARCHAR | pending / processing / completed / rejected |
| created_at | TIMESTAMP | 创建时间 |

### learn_routine_observations（观察记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| routine_id | UUID | 关联的 routine |
| session_id | VARCHAR | 触发会话 |
| matched | BOOLEAN | 是否匹配成功 |
| created_at | TIMESTAMP | 观察时间 |

## 经验蒸馏流程（Distiller）

```
一轮对话结束
    │
    ▼
┌─────────────────────────────────┐
│ 1. 提取对话片段                   │
│    用户问题 + Agent SQL + 执行结果 │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 2. LLM 蒸馏                      │
│    "从这段对话中提炼出通用规则"    │
│    输出: pattern + content        │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 3. 三层防线验证                   │
│    语法校验 → 执行验证 → 用户确认  │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 4. 评分分类                      │
│    SKIP / WRITE_NEW / MERGE /    │
│    REPLACE / TTL_SHORT           │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 5. 写入/更新 learned_routines    │
│    同时写入审计日志               │
└─────────────────────────────────┘
```

## 学习经验如何注入 Agent 决策

```
用户提问: "查一下签收率"
    │
    ▼
┌─────────────────────────────────┐
│ 检索匹配的 routines               │
│                                 │
│ SELECT * FROM learned_routines   │
│ WHERE status = 'active'          │
│ AND pattern MATCHES '签收率'     │
│ ORDER BY score DESC, hit_count   │
│ LIMIT 5                          │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 注入 System Prompt               │
│                                 │
│ "## 学习经验                       │
│  - 签收率 = COUNT(CASE WHEN       │
│    status='已签收' THEN 1 END)    │
│    * 100.0 / COUNT(*)            │
│  - 这个公式经用户确认正确"        │
└──────────────┬──────────────────┘
               │
               ▼
         Agent 参考经验生成 SQL
```

### Agent 可以选择忽略

学习经验是"参考"而非"强制"。Agent 在以下情况会忽略：
- 用户明确指定了不同的计算方式
- 经验与当前数据源不匹配
- 经验的 score 过低（可信度不足）
