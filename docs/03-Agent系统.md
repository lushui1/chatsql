# 03 — Agent 系统

## 主 Agent 职责

ChatSQL 的核心是一个 **单 Agent 系统**，负责将用户的自然语言问题转化为数据查询结果。

### 决策流程

```
用户提问
    │
    ▼
┌─────────────────────────────┐
│ 1. 意图识别                    │
│    用户想查什么？               │
│    - 简单查询 → 直接生成 SQL    │
│    - 复杂分析 → 先调 planning   │
│    - 模糊问题 → 调 clarification│
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 2. 表选择                     │
│    从 System Prompt 的元数据中  │
│    选择目标表和字段             │
│    - 单表查询 → 直接 SELECT    │
│    - 多表关联 → 生成 JOIN      │
│    - 跨库查询 → 多条 SQL 分别跑 │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 3. SQL 生成                   │
│    根据目标数据源的方言生成 SQL  │
│    - MySQL → backtick 引用     │
│    - ClickHouse → 特有函数     │
│    - Hive → DATEDIFF 语法      │
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│ 4. 结果展示                   │
│    根据数据特征选择展示方式      │
│    - 趋势数据 → 折线图          │
│    - 占比数据 → 饼图            │
│    - 对比数据 → 柱状图          │
│    - 明细数据 → 表格            │
│    - 单值数据 → 文本            │
└─────────────────────────────┘
```

### 核心职责

| 职责 | 说明 |
|------|------|
| 意图理解 | 解析用户的自然语言问题，判断查询意图 |
| 表/字段选择 | 从元数据中选择正确的表和字段 |
| SQL 生成 | 根据方言生成可执行的 SQL |
| 结果解读 | 用自然语言描述查询结果 |
| 展示选择 | 根据数据特征选择图表类型 |
| 异常处理 | SQL 报错时分析原因并重试或追问 |

## 工具链

ChatSQL 内置 4 个工具，Agent 通过 function_call 按需调用：

### 1. planning（分析规划）

**触发条件**：复杂问题、多步骤查询、需要拆解的问题

**输入参数**：
```json
{
  "overview": "总体分析概述",
  "steps": [
    {"id": 1, "title": "第一步：确定时间范围"},
    {"id": 2, "title": "第二步：按分组聚合"},
    {"id": 3, "title": "第三步：排序取 Top N"}
  ]
}
```

**作用**：
- 将复杂问题拆解为可执行的步骤
- 向用户展示分析思路，增加透明度
- 帮助 Agent 自己理清查询逻辑

### 2. smartbot_chart（图表生成）

**触发条件**：需要可视化展示的查询结果

**输入参数**：
```json
{
  "type": "line",
  "title": "近7天订单量趋势",
  "xAxis": ["06-15", "06-16", "06-17", "06-18", "06-19", "06-20", "06-21"],
  "series": [
    {"name": "分拣中心A", "data": [120, 132, 101, 134, 90, 230, 210]},
    {"name": "分拣中心B", "data": [220, 182, 191, 234, 290, 330, 310]}
  ]
}
```

**支持的图表类型**：
| type | 适用场景 | 示例 |
|------|---------|------|
| `bar` | 对比数据 | 各网点订单量对比 |
| `line` | 趋势数据 | 近7天订单量走势 |
| `pie` | 占比数据 | 各渠道订单占比 |
| `column` | 分类统计 | 按状态统计包裹数 |
| `table` | 明细数据 | 订单列表详情 |

### 3. ask_clarification（追问）

**触发条件**：用户问题模糊、信息不足、有歧义

**输入参数**：
```json
{
  "question": "您想查的是最近7天还是最近30天的订单数据？"
}
```

**作用**：
- 避免 Agent 猜测用户意图生成错误 SQL
- 提供选项让用户明确需求
- 提高查询准确率

### 4. propose_subscription（订阅建议）

**触发条件**：用户反复查询同一类数据、需要定期监控

**输入参数**：
```json
{
  "message": "建议每天早上 9:00 自动推送昨日各分拣中心的订单汇总"
}
```

**作用**：
- 识别用户的重复查询模式
- 建议创建定时推送，减少重复操作

## fast 模式 vs think 模式

| 维度 | fast 模式 | think 模式 |
|------|----------|-----------|
| 速度 | 快（1-3秒） | 慢（5-30秒） |
| 模型 | 轻量模型（如 gpt-4o-mini） | 重型模型（如 gpt-4o / deepseek-r1） |
| 规划 | 跳过 planning，直接生成 SQL | 先调 planning 分析再执行 |
| 适用 | 简单查询、单表、明确问题 | 复杂分析、多表关联、模糊问题 |
| Token 消耗 | 低 | 高 |

**切换方式**：前端侧边栏顶部 fast/think 按钮，或请求 Header `X-SmartBot-Mode`

## System Prompt 动态注入

Agent 的 System Prompt 不是固定的，而是根据上下文动态组装：

```
System Prompt = 基础指令
              + 数据源方言规则
              + 表元数据（表名/字段/类型/注释/采样数据）
              + 学习经验（匹配的 routines）
```

### 注入内容示例

```
你是一个数据分析助手，根据用户的问题生成 SQL 查询并解释结果。

## 当前数据源
类型: MySQL
方言: MySQL

## 可用表结构

### orders（物流订单）
| 字段 | 类型 | 说明 |
|------|------|------|
| order_id | VARCHAR | 订单号 |
| status | VARCHAR | 状态：已揽收/运输中/已签收 |
| sort_center | VARCHAR | 分拣中心编码 |
| create_time | TIMESTAMP | 创建时间 |
| amount | DECIMAL | 金额 |

### sort_center（分拣中心）
| 字段 | 类型 | 说明 |
|------|------|------|
| center_id | VARCHAR | 中心编码 |
| center_name | VARCHAR | 中心名称 |
| region | VARCHAR | 所属区域 |

## 学习经验
- 用户常问"订单量"时，通常指 COUNT(DISTINCT order_id)
- "最近7天"通常用 WHERE create_time >= DATE_SUB(NOW(), INTERVAL 7 DAY)

## SQL 方言规则
- 使用反引号 `` ` `` 引用标识符
- 日期函数: DATE_FORMAT / DATE_SUB / NOW()
- 分页: LIMIT offset, count
```

### 动态注入的好处

1. **元数据变化时自动适应** — 新增表/字段无需改代码
2. **数据源切换时方言自动切换** — 从 MySQL 切到 ClickHouse，SQL 语法自动变
3. **学习经验持续积累** — 越用越准
4. **不同会话可以不同数据源** — 每个会话独立的 System Prompt
